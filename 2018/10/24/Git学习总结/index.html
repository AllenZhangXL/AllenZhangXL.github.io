Git 学习总结

Posted on 2018-10-24 | In Git

在学习了廖雪峰老师的Git讲解之后，我知道了Git的发展过程，以及Git的基本用法。更加深入的明白了GIthub，Git 之间的关系。
下面是我在学习廖雪峰老师的课程时，自己的一些总结。

  1. 创建版本库
    输入下面命令创建版本库：
    $mkdir nameofku
    $cd nameofku
    $pwd

  2. 将这个目录变成可以管理的仓库：
    $git init

  3. 添加文件到仓库：
    $git add nameOfFile

  4. 提交
    $git commit -m "备注操作"

每次对文件的修改都要执行 add 和 commit 操作
$git status 显示仓库当前状态
$git diff nameOfFile 显示文件修改前后的不同

##Git 版本回退

  • $git log 查看所有 commit -m 的信息（修改时间，内容，被谁修改）

git 版本退回是根据 commit id 进行， $git log 可以查看commit id

##工作区和暂存区

工作区使我们电脑可以看到的目录

暂存区使我们看不到的 .git 隐藏目录，是git的版本库
git版本库中存了很多文件，主要是Stage（暂存区），以及git为我们自动创建的第一个分支master，以及指向master的一个指针HEAD。
做任何修改的时候，先要git add 将修改提交到暂存区，
第二步， git commit 将暂存区的内容提交到当前分支。

##撤销修改

  • 场景1：改乱了工作区的某个文件内容，想直接恢复之前的内容， 使用命令 $git checkout --fileName
  • 场景2：改乱了工作区的文件内容，而且还提交到了暂存区， 分为两步：
    1. $git reset HEAD <fileName> ，回到场景1，
    2. 执行场景1的操作。
  • 场景3：已经将错误的文件提交到版本库时，需要执行上面版本回退的操作。

##删除文件

删除也是一个修改操作，在本地文件管理器中使用 rm 删除文件
这是工作区和版本库不一致， git status 查看会告诉那些文件不一致、
如果确认删除， 使用 git rm fileName & git commit
误删，则使用 git checkout --fileName 恢复

##远程仓库GITHUB

GITHUB是远程仓库，用于备份和与其他人分享。

  1. 注册github账号
  2. 创建SSH Key
    打开git bash–> $ssh-keygen -t rsa -C "youremail@example.com"
    一路回车，在.ssh 目录中找到 id_rsa（私钥） 和 id_rsa.pub(公钥)。
  3. 登录github，account settings–> ssh keys 添加 SSH key 将公钥复制，并保存。

##添加仓库

  1. 登录github –> ceate a new repo
  2. 本地仓库下
    $git remote add origin git@github.com:username/repoName.git
  3. 推送本地内容
    $git push -u origin master
    将本地内容推送到github，因为是第一次推送，所以要加 -u 参数。
    之后的推送可以 $git push origin master

##从远程库克隆

如何将很多线上项目搬到本地：克隆
使用命令：
$git clone git@github.com:uesrname/repo-name.git
将远程仓库克隆到本地。
git支持多种协议，默认的git：//使用ssh， 也可使用https等协议。
https速度慢，每次推送都要输入口令，
ssh支持原生git协议，速度最快。

##创建于合并分支

创建分支dev，并切换到分支dev：
$git checkout -b dev -b 表示创建并切换
$git branck 查看当前分支
$git checkout branchName 切换分支
$git merge branch1 将 branch1 的内容合并到 当前分支上
$git branch -d branchName 删除分支

##分支管理策略

不使用Fast forward 模式， git 会在merge时生成新的commit，这样就可以从分支历史记录上看出分支的信息。
$git merge --no-ff -m "merge with no-ff" BranchName

##Bug分支

Git可以为每个Bug创建一个解决branch，解决后，merge，并删除临时的branch。
$git stath 将现在的工作现场保存，等修改完bug之后回到现场继续工作。

  1. 切换到bug分支 -> $git checkout bug1Branch
  2. 创建并切换到临时解决bug1的分支 -> $git checkout -b issue-101
    在当前分支上解决bug，然后合并
  3. $git merge --no-ff -m"merge bug fix 101" issue-101
  4. 切换回之前的工作的分支 $git checkout workBranch
    查看工作现场 git stash list
  5. 恢复工作现场使用 $git stash pop
    也可以使用 $git stash apply 恢复现场但不删除保存的现场， 需要$git stash drop 删除

##Feature 分支

软件开发中，为了保证主分支，不被其他新功能分支的试验性代码搞乱，我们在添加新功能的时候，就要创建新的分支。
利用上面说到的创建分支的命令，创建功能分支。
如果在功能完成，合并时，不需要新的功能，则删除功能分支：
$git branch -d Branch1
$git branch -D Branch1强制删除

##多人协作

多人协作的工作模式是这样：

  1. 首先，试图视同$git push origin branch-name推送自己的修改
  2. 如果推送失败，可能因为远程分支比本地更新，需要$git pull 试图合并：
  3. 如果合并有冲突，则需要手动解决，并在本地提交；
  4. 没有冲突或者解决掉冲突后，再用git push origin branch-name推送就能成功！

如果 git pull 提示 no tracking information 说明本地分支和远程分支的链接关系没有建立， 使用命令 $git branch --set-upstream-to=origin/Branch-name Branch-name

##标签管理

发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本。将来无论什么时候，取某个标签的版本，就是把那个打标签的时刻的历史版本取出来。所以，标签也是版本库的一个快照。
Git的标签虽然是版本库的快照，但其实他就是指向某个commit 的指针，因此创建和删除标签都是瞬间的。
tag实际就是一个让人容易记住的有意义的名字，他跟commit绑定在一起。

  1. 切换到需要打标签的分支 $git checkout branch1
  2. 打标签 $git tag v1.0
  3. 查看标签 $git tag

如果要对之前的 commit 打标签， 则需要找到对应得commit id 然后打标签
比如对 f12345 打标签
$git tag v0.9 f12345

还可以创建带有说明的git tag
$git tag -a v0.1 -m "version 0.1 released" f12345

使用命令$git show <tagname> 看到文字说明

###操作标签

命令git push origin <tagname>可以推送一个本地标签；

命令git push origin --tags可以推送全部未推送过的本地标签；

命令git tag -d <tagname>可以删除一个本地标签；

命令git push origin :refs/tags/<tagname>可以删除一个远程标签。
